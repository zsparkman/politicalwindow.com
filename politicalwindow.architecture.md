## 1. Purpose

The user-facing web frontend for Political Window — a political-advertising
intelligence product. Static HTML/CSS/JS hosted on GitHub Pages at
`politicalwindow.com`. It is a **multi-page** static site (not a SPA in the
routing sense) whose pages share a common style sheet and pull live data at
load time from `politicalwindow-api` (candidate / state / ballot / poll data)
and `ratewindow-api` (FCC ad-rate intelligence, candidate-tracker spend, LUR
analysis, PDF proxying, auth).

> **Inconsistency:** `CLAUDE.md` (this repo) describes the frontend as a
> "single file: `index.html` contains all HTML, CSS, and JavaScript inline"
> and rule #1 reads "`index.html` stays a single self-contained file — no
> external JS or CSS files." The committed tree contradicts both: it
> contains seven HTML pages and a shared `style.css`. Treat the
> committed tree as the source of truth and CLAUDE.md as out of date on
> this point.

## 2. Tech Stack

- **Runtime:** none on the server — pure static hosting.
- **Build step:** none. No `package.json`, no bundler, no transpiler. Files
  are served verbatim by GitHub Pages.
- **CDN-loaded libraries** (referenced from `index.html`):
  - **MapLibre GL JS 4.7.1** (unpkg) — vector basemap rendering. Falls back
    to a hand-rolled tile-grid cartogram if the CDN fails (CLAUDE.md
    "External CDN Dependencies").
  - **CARTO Dark Matter** basemap tiles (no API key).
- **Fonts:** Bebas Neue (display), IBM Plex Mono (data), DM Sans (body) —
  loaded via Google Fonts.
- **Hosting:** GitHub Pages with custom domain. `CNAME` contains
  `politicalwindow.com`. No `.github/workflows/` directory exists, so
  deployment uses GitHub Pages' default "Deploy from branch" mode (push
  to the configured branch → published).

## 3. Entry Points and Boot Sequence

There is no shared entry point. Each top-level page is a standalone HTML
document that links to `style.css` and contains its own JS. The flagship
page is `index.html` (the map + state-detail experience). On boot,
`index.html`:

1. Defines hard-coded API base URLs at `index.html:4035-4036`:
   - `API_BASE  = 'https://api.politicalwindow.com'`
   - `RATES_API = 'https://rates.politicalwindow.com'`
2. Calls `loadLiveData()` (defined per CLAUDE.md function table) which
   fans out parallel `fetch()` calls (`index.html:4042-4045`):
   - `${API_BASE}/api/states`
   - `${API_BASE}/api/candidates`
   - `${API_BASE}/api/ballot-measures`
   - `${API_BASE}/api/polls`
3. Populates the runtime data structures (`SM`, `CANDS`, `BALLOT`, `POLLS`
   — see CLAUDE.md "Data Structures").
4. Pulls a US-states GeoJSON from a third-party raw GitHub URL
   (`index.html:1457`: `raw.githubusercontent.com/PublicaMundi/MappingAPI`).
5. Initializes MapLibre via `initMap()`; on CDN failure falls back to the
   tile-grid `buildMap()` rendering.
6. Wires per-state interactions (`pick()`, `mapSelectState()`,
   `wireDetailTabs()`) and the embedded login overlay (see Section 8).

Other pages (`candidate-tracker.html`, `lur.html`, `rates.html`,
`explorer.html`, `public-file.html`, `admin.html`) follow the same pattern:
inline JS, hit one or both APIs, render.

## 5. Data Model

No direct database access. The frontend's "data model" is the in-memory
shape that `loadLiveData()` produces from API responses
(`index.html:4035-...`, see CLAUDE.md "Data Structures"):

- `SM` — `{ TX: { pWindow, rWindow, dWin, dPrim, dRun, dRunW, status }, ... }`
  (state metadata; status drives map color)
- `CANDS` — `{ TX: [{ race, list:[{n, p, s, note, d}] }] }` (candidates by
  state; `d` is the House district number)
- `BALLOT` — ballot measures grouped by state (from `/api/ballot-measures`)
- `POLLS` — `{ [state_abbr]: { [race]: [{ pollster, date, sample, moe, results }] } }`
- `glMap`, `statesGeoJSON`, `DISTRICT_CACHE`, `_financeCache`,
  `candTabState`, `_currentStateAbbr`, `_hoveredDistrictId`,
  `DISTRICT_COUNTS` — UI-state caches; see CLAUDE.md for full descriptions.

CLAUDE.md rule #3 forbids `localStorage`, `sessionStorage`, and cookies.

> **Inconsistency:** the embedded mini-login overlay at the bottom of
> `index.html:4434` violates that rule — on a successful `POST /auth/login`
> it calls `localStorage.setItem('pw_token', res.d.token)` and
> `localStorage.setItem('pw_user', res.d.email)`. Either the rule is
> stale or this code is. Flag for the maintainer.

## 8. Authentication and Authorization

The frontend itself has no auth; pages render and degrade gracefully on
401/403. Authentication is only required for `ratewindow-api` pro/admin
endpoints, and the JWT round-trip is owned by the **mini-login overlay**
embedded as an IIFE at the bottom of `index.html:4434`:

1. User submits email + password to `https://rates.politicalwindow.com/auth/login`.
2. On 200, the JWT and email are stored in `localStorage` (`pw_token`,
   `pw_user`) and the page reloads.
3. Subsequent fetches read `localStorage.pw_token` and send
   `Authorization: Bearer <token>`. The `ratewindow-api` auth middleware
   resolves the JWT to a synthesized `pro`-tier API-key context (see
   `ratewindow-api/middleware/auth.js`).

Registration, password change, key generation, and user disabling all
flow through `admin.html` against the admin-only `/api/invites*` and
`/api/keys` routes on `ratewindow-api`.

There is no client-side role/permission model: tier gating is enforced
server-side and the UI just hides or grays features that 403.

## 12. Known Quirks and Gotchas

- **`index.html` is enormous** (~221 KB / ~4500+ lines of inline JS) and
  the `loadLiveData` fan-out plus MapLibre boot all run in the same
  script. Any blocking error mid-script can leave the map and detail
  panel partially wired.
- **MapLibre fallback** is a real fallback path (`buildMap()` tile-grid
  cartogram). Code that touches `glMap` must null-check.
- **`candTabState` persists per-state across re-entry** (CLAUDE.md), so
  switching away from a state and back preserves Candidates-tab UI
  selections (office, expanded parties, drilled-in district).
- **At-large districts**: only AK, DE, ND, SD, VT, WY are labelled `AL`.
  `DISTRICT_COUNTS` is the single source of truth for "is this state
  at-large?" (CLAUDE.md).
- **Active-district illumination is unconditional on House-bearing
  states** (Follow-up #2, 4/15) — every loaded CD lights up regardless
  of tracker spend coverage. `getCandidatesFor` is no longer the gate.
- **Station-display convention**: dropdown `value` keeps the raw
  callsign/PSID; only `textContent` shows the human label. Never change
  the field function in `updateOptionStates` to return the brand string.
- **Multi-page reality vs. CLAUDE.md "single file" claim** — see the
  inconsistency noted at the top of this document.
- **`localStorage` use vs. CLAUDE.md "no client-side persistence" rule**
  — see Section 5.
- **`rate_lines` is incomplete relative to `political_invoices`** (audited
  2026-05-12 against KTBY: invoices $167,071 / 20 rows; rate_lines
  $67,839 / 257 rows — **41% gross coverage**). Many invoices land in
  `political_invoices` from header totals when their per-line spot
  detail isn't parseable. **Rule for any view layer:** sum
  `political_invoices.gross_amount` for $ totals, station rollups,
  gap calculations, share %, and LUR-exposure-related numbers.
  Consult `rate_lines` only when you need daypart, broadcast-week,
  or per-spot-rate granularity, and label the chart so the viewer
  understands they're looking at the line-level subset, not the full
  invoiced gross. The `w/coastal.html` weekly trend chart was
  initially built off `rate_lines` and visually undercounted Coastal
  by ~$100K — fixed by switching to invoice-level aggregation
  bucketed via the `weekStart()` helper. Don't repeat the mistake.
- **`political_invoices.fcc_file_url` is NULL footprint-wide** (audited
  2026-05-12 against the Coastal callsign set: 0 of 114 rows
  populated). The column exists but has never been written. To
  surface "View PDF" links in any future viewport, build a
  client-side cross-reference map from `/api/lines` responses keyed
  by `station|order_number` — `/api/lines` constructs a working FCC
  public-file URL at query time
  (`'https://publicfiles.fcc.gov/api/manager/download/' || ff.folder_id || '/' || ff.file_manager_id || '.pdf'`,
  see `ratewindow-api/index.js:721`). The `pdfUrlFor(inv)` helper
  in `w/coastal.html` is the canonical pattern; ~93% PDF coverage
  via this fallback. The remaining ~7% are invoices whose
  `rate_lines` rows weren't extractable — render those rows with a
  "—" placeholder and a hover explanation, never a fabricated link.

## 14. Workspaces (`/w/<slug>`)

Introduced 2026-05-12 EOD. **Workspace** is the right architectural
abstraction for the per-customer dashboard surface. The first
workspace (Coastal TV at `/w/coastal`) is the only concrete instance
today; this section documents the pattern so the second, third, and
nth workspace land consistently.

### 14.1 Concept

A workspace is a **scoped lens on the underlying invoice / line / LUR
data lake, parameterized by persona.** Same data source, different
narrative framing depending on who the workspace is for. URL
namespace is `/w/<slug>` (path-based, not subdomain — subdomain
rewrite stays available as a CDN-layer customization for white-glove
customers; not implemented).

Today's implementation is **one self-contained HTML file per
workspace** under `politicalwindow.com/w/`. The file is a static-host
SPA that authenticates against `ratewindow-api`'s existing JWT flow,
fans out read-only API calls, and aggregates client-side. No backend
changes were required to ship the first workspace; same is true for
the next several.

### 14.2 Workspace types

| Type | Scope key (filter on `political_invoices`) | Persona | Headline metric | Status |
|---|---|---|---|---|
| `ownership_group` | set of `station_call_sign` | Group exec, sales VP | Gap $ vs competitors | **Live** — Coastal at `/w/coastal` |
| `agency` | set of `agency_name` (via `political_invoices.agency_name`) | Agency principal, media director | $ placed across stations | v2 — planned |
| `campaign` | set of `political_entity_id` | Campaign manager, treasurer, FEC compliance | $ spent + LUR exposure (refund opportunity) | v2 — planned |
| `custom` | arbitrary saved filter (HogQL-style or structured) | Analyst, consultant, journalist | configurable | v2 — planned |

### 14.3 Workspace config schema

Today the config is the `GROUP` constant at the top of
`w/<slug>.html`. When a second workspace type is built, the constant
becomes a **typed `WORKSPACE` config** (TypeScript-like contract;
JS-only at runtime):

```
WORKSPACE = {
  /* Common to all types */
  type:             'ownership_group' | 'agency' | 'campaign' | 'custom',
  slug:             'coastal-tv',         /* matches URL: /w/<slug> */
  display_name:     'Coastal Television Broadcasting Group',
  short_name:       'Coastal TV',         /* used in headlines, breadcrumbs */
  cycle:            2026,
  branding?:        { logo_url, primary_color },  /* future */

  /* Type-discriminated scope */
  scope: <one of>:

    /* ownership_group — current, hardcoded today as `stations[]` */
    {
      stations: [
        { callsign:  'KTBY',
          dma_label: 'Anchorage, AK',
          dma_db:    'ANCHORAGE',         /* MUST match political_invoices.station_market UPPERCASE */
          network:   'FOX',
          state:     'AK' },
        ...
      ],
      peer_share_pct: 22                  /* benchmark for the share-donut tick */
    }

    /* agency — v2 */
    {
      agency_names: ['BUYING TIME LLC', 'CANAL PARTNERS MEDIA', ...],
      states:       ['CA','TX','FL']     /* optional scope narrowing */
    }

    /* campaign — v2 */
    {
      entity_ids: ['<uuid>', '<uuid>'],   /* political_entities.political_entity_id */
      cycle:      2026
    }

    /* custom — v2 */
    {
      filter: '<HogQL-style WHERE expression>',
      preview_max_rows: 1000
    }
}
```

The `dma_db` field for ownership-group stations is a **hard
contract**: it must equal the `political_invoices.station_market`
value verbatim (UPPERCASE FCC-format, e.g. `ANCHORAGE`,
`FARGO-VALLEY CITY`, `LAFAYETTE, IN`). Mismatches show as "No
coverage" tags on the market scoreboard — the dashboard does not
auto-correct, deliberately (see ground rule "no placeholder data in
the live view").

### 14.5 Data resolution per scope-type

Each scope-type implies a different fanout strategy against
`ratewindow-api`:

| Type | Endpoints called | Notes |
|---|---|---|
| `ownership_group` | `/api/invoices?state=<st>` × distinct states; client-side filter to `dma_db` set; `/api/lur?callsign=` × owned callsigns; `/api/lines?station=` × owned callsigns | Current implementation. ≤9 invoice calls, 22 LUR calls, 22 line calls for Coastal |
| `agency` | Would need `/api/invoices?agency=<name>` filter (not currently supported — small API addition needed) | Or fall back to `/api/invoices?state=` then client-side filter on `agency_name` |
| `campaign` | `/api/invoices?entity_id=<uuid>` × N entity_ids (already supported) | Cleanest — scope key is exactly the existing API filter |
| `custom` | `/api/lines` with arbitrary filter chain (already supports state / advertiser / party / office / daypart_code / week range) | Requires a query builder UI |

Two cross-cutting data-resolution patterns to reuse for any workspace
that needs them (both documented in §12 Quirks):

- **`weekStart(dateStr)`** — buckets invoices by ISO Monday of
  `flight_start` for weekly time series. Use this, not
  `rate_lines.week_of` summation, for any chart that needs to tie
  back to invoiced gross totals.
- **`pdfByOrder` map + `pdfUrlFor(inv)` helper** — resolves a
  click-to-view PDF URL for invoices via the `/api/lines`
  cross-reference, since `political_invoices.fcc_file_url` is NULL
  footprint-wide.

### 14.7 Onboarding

**Today (one HTML file per workspace).** From the README:

1. Copy `w/coastal.html` → `w/<new-slug>.html`.
2. Edit the `GROUP` constant at the top: `slug`, `display_name`,
   `short_name`, `cycle`, `peer_share_pct`, `stations[]` (each
   station: `callsign`, `dma_label`, `dma_db` UPPERCASE matching
   the DB, `network`, `state`).
3. Push to `main`. GH Pages serves the new workspace at
   `politicalwindow.com/w/<new-slug>` immediately.
4. Issue an invite token via the admin console (`/admin`). The
   workspace inherits the existing `ratewindow-api` JWT auth flow.

**v2 (typed `WORKSPACE` config, still per-file).** Same workflow,
but the constant is type-discriminated so the engine renders the
appropriate views per workspace type. Onboarding form factor stays
copy-and-edit; the constant just gets richer.

**v3 (DB-backed workspaces with admin UI).** `workspaces` table +
CRUD endpoints + `workspace_users` ACL table on `ratewindow-api`.
Admin UI for creating / editing workspaces. Onboarding becomes a
form, not a file edit. `w/<slug>.html` becomes a thin shell that
fetches its config from `/api/workspaces/<slug>` instead of reading
a hardcoded constant.

### 14.8 Roadmap

| Phase | Scope | Trigger |
|---|---|---|
| **v1 (today)** | One workspace type (ownership_group), one workspace (Coastal), hardcoded `GROUP` constant per file | — |
| **v2** | Add agency / campaign / custom workspace types; promote `GROUP` to typed `WORKSPACE` config; refactor view library to declare scope-type compatibility; add per-type Overview / Competition framings; small `ratewindow-api` addition for `/api/invoices?agency=` filter | First non-owner customer signs |
| **v3** | DB-backed workspaces (`workspaces` table on `ratewindow-api`), CRUD endpoints, per-workspace ACL (`workspace_users`), admin UI for create/edit, thin-shell `w/<slug>.html` reading config from API | ~5–10th customer; manual file maintenance becomes the bottleneck |
| **v3+** | Subdomain rewrite at the CDN layer (`<slug>.politicalwindow.com` → `/w/<slug>`); workspace-template marketplace; cross-workspace consultants (one user → multiple workspace ACLs) | White-glove enterprise tier |
